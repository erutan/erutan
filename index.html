a bit about me

I'm one of those annoying folks that really cares about the semantic usage of HTML elements, and I build thoughtful reusable CSS (nothing quite like making some branding changes to a shared custom library, have multiple applications pull them in as a git submodule, and not spend much time cleaning up visual glitches). It's pretty common for me to swap between a few other hats on a gig: research and analysis, ux/persona development, information/site architecture, content strategy, copy writing, site configuration, and non-critical sysadmin.

I personally find it much more interesting to be a jack of all trades, and it ends up fitting well into small teams. Often times there isn't a need to have a dedicated person for each role, and handling multiple roles ends up saving time, money, and decreases the chance of miscommunication. Once you're far enough (subjective) into a visual design you can switch to developing in browser vs creating yet more artifiacts to toss away at the end.

web application prototyping

I usually shift quickly into designing with code front-end prototyping once a general direction is set via graphical comps. The problems tend to be more intellectually stimulating, the technology cleaner, and the contracts less rushed than building yet another brochure site.

Some examples of work for an internal team at UCLA are the UI for their Shibboleth Single Sign-On System and the initial release of their online self-service management tool.

content management systems

I've been focusing on Drupal recently, while it's a little clunkier to theme than some of the other options, you can chain together modules to do some pretty powerful things relatively simply. I was lead front-end developer for a redesign of UCSF.edu built on Drupal 7, have given community talks to Drupal groups in the Bay Area, and have done content strategy, migration, site-building, and theming for multiple smaller projects.

On the other side of the spectrum, a lot of websites don't need the overhead of a modern database driven CMS. I've built out a Jekyll internal documentation system for Looker and most of my own personal tinkering is in this space.
